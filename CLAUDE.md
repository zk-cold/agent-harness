# CLAUDE.md

## Invariants

1. **All modifications to `.md` files in this project must be reviewed by 2 critic agents before commit.** Each critic returns APPROVE or REJECT (with reasons). There is no "approval with comments" — if changes are needed, the critic must REJECT. On rejection, the lead agent has discretion to fix and resubmit or escalate to dev.
2. **Invariants are best captured as tests, not text.** Where possible, encode invariants as automated tests rather than prose rules. Text invariants are a last resort when a test is infeasible.
3. **For every work request, the lead agent must invoke `/enhance-harness` or `/new-sdlc` before beginning execution. If the agent cannot determine which skill applies, it must ask the user to disambiguate.** Violation: the agent produces deliverables (files, commits, mission.md) for a work request without having first invoked one of the two routing skills, or proceeds despite ambiguity without asking the user. (Text-only: no instrumentation exists to log skill invocations or assert invocation order; the harness currently has no test infrastructure for agent conversation behavior. A test becomes feasible when the harness gains session-level observability.)
4. **Existing tests are invariants.** Modifying or deleting existing test code is an invariant change. Adding new tests is not. Violation: a commit diff modifies or deletes existing test code without following the invariant-change process (2-critic review, normal flow). (Text-only: no CI or pre-commit hook exists to detect test-file modifications automatically. A test becomes feasible when the harness gains a pre-commit check that flags diffs touching existing test files.)
