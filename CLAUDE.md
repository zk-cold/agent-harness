# Invariants

## Invariants

1. **For every work request, the lead agent must invoke `/enhance-harness` or `/new-sdlc` before performing bootstrap discovery or creating a worktree. If the agent cannot determine which skill applies, it must ask the user to disambiguate.** Violation: the agent performs bootstrap discovery or creates a worktree before invoking one of the two routing skills, produces deliverables (files, commits, mission.md) for a work request without having first invoked one of the routing skills, or proceeds despite ambiguity without asking the user. (Text-only: no instrumentation exists to log skill invocations, assert invocation order, or detect whether bootstrap discovery preceded routing. A test becomes feasible when the harness gains session-level observability.)
2. **After routing to the applicable skill, the lead agent must perform bootstrap discovery against the applicable repo root. For `enhance-harness`, the applicable repo root is the harness repo root; the lead agent must check it before any interview or execution work begins. For `new-sdlc`, the applicable repo root is the target repo root; the lead agent must identify the target repo root during the interview before any repo-specific analysis begins, then immediately check it for root artifacts and worktree state before any further interview or execution work proceeds. If `handoff.md` or `mission.md` exists at the applicable repo root, the lead agent must stop and ask the dev how to proceed before continuing. During the interview, the lead agent must check for worktrees under the applicable `.claude/worktrees/` directory: if any worktree contains a `handoff.md` that the lead agent judges to be related to the current request, the lead agent must surface that handoff to the dev and ask how to proceed; otherwise, existing worktrees must be ignored. If the lead agent will treat the current prompt as fresh work instead of resuming a related `handoff.md`, it must confirm with the dev before discarding or replacing that handoff. The lead agent must create a worktree in the applicable repo before any execution work begins. All runtime artifacts (`handoff.md`, `mission.md`) must be written to the worktree root, never the applicable repo root. Whenever the lead agent spawns a non-lead agent after completing this bootstrap check, the spawned-agent prompt must include the exact sentence stem `You are a non-lead agent.`. Any spawned non-lead agent must follow that prompt and must not perform bootstrap discovery.** Violation: the agent begins repo-specific analysis before identifying the target repo root and checking it for root artifacts (for `new-sdlc`), continues despite root-level runtime artifacts without dev direction, ignores a worktree handoff that the lead agent judges to be related to the current request, discards or replaces a related `handoff.md` while treating the prompt as fresh work without dev confirmation, creates a worktree in the wrong repo, writes `handoff.md` or `mission.md` to the applicable repo root instead of the worktree root, fails to include in a spawned non-lead prompt the exact sentence stem `You are a non-lead agent.`, or a spawned non-lead agent performs bootstrap discovery. (Text-only: no instrumentation exists to detect which repo root was inspected, whether bootstrap discovery preceded repo-specific analysis, whether a related worktree handoff was surfaced to the dev, or whether runtime artifacts were written to the correct location. A test becomes feasible when the harness gains session-level observability.)
3. **Whenever `/enhance-harness` or `/new-sdlc` requires critic review, the lead agent must complete that review before continuing. If spawning a required critic agent is blocked because the environment requires explicit user approval, the user declines, or the critic tool is otherwise unavailable, the lead agent must stop, rewrite `handoff.md` with the blocker, and ask the dev for direction. The lead agent must not skip that review phase, continue execution, or proceed to Cleanup past it.** Violation: the agent skips a required critic-review phase, treats missing critic availability as implicit approval, continues execution or cleanup without the required review, or fails to record and surface the blocker when critic spawning is unavailable. (Text-only: no instrumentation exists to log required review phases, critic-spawn blockers, or whether execution advanced past an unmet review gate. A test becomes feasible when the harness gains session-level and tool-invocation observability.)
4. **Until the active skill's required completion-review phase approves the implementation, the lead agent and any spawned dev agent must keep the implementation uncommitted in the worktree. Before completion review, the lead agent must merge the latest applicable repo-root branch state into the current worktree and reconcile the implementation against that merged state. For `enhance-harness`, the applicable repo-root branch state is the current `main` branch head of the harness repo root. For `new-sdlc`, the applicable repo-root branch state is the integration branch named by the target repo's governing artifacts; if those artifacts name none, it is the branch from which the worktree was created. A merge is non-trivial when it is neither `Already up to date.` nor a clean fast-forward with no merge commit and no manual conflict resolution. Completion-review critics must review the live uncommitted diff plus the required runtime verification artifacts for that merged state. If that pre-review merge, or any cleanup-phase merge attempted before the worktree is removed, is non-trivial, the lead agent must reset the mission to the applicable completion-review phase, re-verify the acceptance criteria against the merged state, and obtain fresh completion-review approval before creating additional commits or removing the worktree. Only after that approval may the lead agent create the commit or commits needed to leave the worktree clean for Cleanup.** Violation: the agent creates a commit before completion-review approval, asks a completion-review critic to review a diff that predates the latest applicable repo-root merge, treats a merge that is not `Already up to date.` and not a clean fast-forward as already approved, removes the worktree after such a merge without resetting to the applicable completion-review phase, or removes the worktree while approved implementation changes remain uncommitted. (Text-only: no instrumentation exists to detect whether the latest applicable repo-root state was merged before completion review, whether a merge was non-trivial under this definition, whether the mission reset to completion review after such a merge, or whether worktree removal happened only after fresh approval of the merged state. A test becomes feasible when the harness gains review-phase and merge-state observability.)
5. **Once a `mission.md` has been approved for the active skill, that approved mission is immutable for the rest of that mission. This applies to both `/enhance-harness` and `/new-sdlc`.** If implementation or review reveals the mission must change, the lead agent must stop the current mission, keep the approved `mission.md` unchanged, and follow `.agent/schemas/abort-protocol.md`, including rewriting `handoff.md` to state that the mission is already aborted and not resumable. The lead agent must not autonomously create a replacement mission. Violation: the agent edits an approved `mission.md`, continues execution or review under a modified mission, fails to record the abort state in `handoff.md`, resumes a mission already marked aborted, or drafts a new mission without first stopping and surfacing blockers to the user. (Text-only: no instrumentation exists to detect post-approval `mission.md` edits, whether the agent aborted when an approved mission became invalid, whether the non-resumable abort state was recorded, or whether it autonomously drafted a replacement mission. A test becomes feasible when the harness gains mission lifecycle observability.)

## Dependencies

- **Python ≥ 3.10** — Required by all scripts under `scripts/`. The `str | None` union type syntax used in the mission linter requires Python 3.10 or later.
- **Git ≥ 2.17** — Required for worktree operations (`git worktree add`, `git worktree remove`) used throughout the harness workflow.

# Considerations

## Considerations

- **Bootstrap check** — Run within Mission Creation after routing, against the applicable repo root. Two subcommands:
  - `python3 -m scripts.bootstrap_check [repo_root]` — Outputs one of: `ROOT_ARTIFACTS`, `MULTIPLE_WORKTREES`, `ONE_WORKTREE_NO_HANDOFF`, `ONE_WORKTREE_ABORTED`, `ONE_WORKTREE_RESUMABLE`, `NO_WORKTREE`. Exits 0 in all cases. Use to detect root artifacts and get an overall worktree-state summary.
  - `python3 -m scripts.bootstrap_check list [repo_root]` — Outputs one line per worktree: `<path> <status>`, where status is `RESUMABLE`, `ABORTED`, or `NO_HANDOFF`. Exits 0 in all cases (outputs nothing when no worktrees exist). Use during the interview to inspect individual worktrees and judge relatedness.
