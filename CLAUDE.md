# CLAUDE.md

## Invariants

1. **On session start, the lead agent must inspect `.claude/worktrees/` and the main project root before any interview or execution work begins. If `handoff.md` or `mission.md` exists at the main project root, the lead agent must stop and ask the dev how to proceed before continuing. Otherwise, if exactly one worktree exists and it contains `handoff.md` at its root, the lead agent must load that `handoff.md` and use it to orient. If multiple worktrees exist, or if exactly one worktree exists but it does not contain `handoff.md`, the lead agent must stop and ask the dev how to proceed before continuing. If no worktree exists and no legacy root runtime artifacts exist, the lead agent must create a worktree before any interview or execution work begins. All runtime artifacts (`handoff.md`, `mission.md`) must be written to the worktree root, never the main project root. If a `handoff.md` exists and the lead agent will treat the current prompt as fresh work instead of resuming that handoff, it must first confirm with the dev before discarding or replacing it. Whenever the lead agent spawns a non-lead agent after completing this session-start check, the spawned-agent prompt must include the exact sentence stem `You are a non-lead agent.` and must explicitly state that the recipient operates only on the role-specific inputs granted by its phase and must not perform bootstrap discovery of `.claude/worktrees/`, the main project root, `handoff.md`, or `mission.md` unless the active phase explicitly requires that bootstrap task for the spawned role. Any spawned non-lead agent must follow that prompt and must not perform bootstrap discovery outside those explicit phase instructions.** Violation: the lead agent begins work on a new request without checking both the worktree directory and the main project root first, continues despite legacy root runtime artifacts or ambiguous worktree state without dev direction, writes `handoff.md` or `mission.md` to the main project root instead of the worktree root, discards or replaces an existing `handoff.md` while treating the prompt as fresh work without dev confirmation, fails to include in a spawned non-lead prompt the exact sentence stem `You are a non-lead agent.` plus the role-specific-inputs and no-bootstrap instructions, or a spawned non-lead agent performs bootstrap discovery without an explicit phase instruction requiring it. (Text-only: no instrumentation exists to detect whether the lead agent checked for a worktree, whether a spawned non-lead agent performed unauthorized bootstrap discovery, whether the lead agent included the required non-lead prompt language, or whether runtime artifacts were written to the correct location at session start. A test becomes feasible when the harness gains session-level observability.)
2. **For every work request, the lead agent must invoke `/enhance-harness` or `/new-sdlc` before beginning execution. If the agent cannot determine which skill applies, it must ask the user to disambiguate.** Violation: the agent produces deliverables (files, commits, mission.md) for a work request without having first invoked one of the two routing skills, or proceeds despite ambiguity without asking the user. (Text-only: no instrumentation exists to log skill invocations or assert invocation order; the harness currently has no test infrastructure for agent conversation behavior. A test becomes feasible when the harness gains session-level observability.)
3. **Whenever `/enhance-harness` or `/new-sdlc` requires critic review, the lead agent must complete that review before continuing. If spawning a required critic agent is blocked because the environment requires explicit user approval, the user declines, or the critic tool is otherwise unavailable, the lead agent must stop, rewrite `handoff.md` with the blocker, and ask the dev for direction. The lead agent must not skip that review phase, continue execution, or proceed to Cleanup past it.** Violation: the agent skips a required critic-review phase, treats missing critic availability as implicit approval, continues execution or cleanup without the required review, or fails to record and surface the blocker when critic spawning is unavailable. (Text-only: no instrumentation exists to log required review phases, critic-spawn blockers, or whether execution advanced past an unmet review gate. A test becomes feasible when the harness gains session-level and tool-invocation observability.)
4. **Until the active skill's required completion-review phase approves the implementation, the lead agent and any spawned dev agent must keep the implementation uncommitted in the worktree. Before completion review, the lead agent must merge the latest applicable repo-root branch state into the current worktree and reconcile the implementation against that merged state. For `enhance-harness`, the applicable repo-root branch state is the current `main` branch head of the harness repo root. For `new-sdlc`, the applicable repo-root branch state is the integration branch named by the target repo's governing artifacts; if those artifacts name none, it is the branch from which the worktree was created. A merge is non-trivial when it is neither `Already up to date.` nor a clean fast-forward with no merge commit and no manual conflict resolution. Completion-review critics must review the live uncommitted diff plus the required runtime verification artifacts for that merged state. If that pre-review merge, or any cleanup-phase merge attempted before the worktree is removed, is non-trivial, the lead agent must reset the mission to the applicable completion-review phase, re-verify the acceptance criteria against the merged state, and obtain fresh completion-review approval before creating additional commits or removing the worktree. Only after that approval may the lead agent create the commit or commits needed to leave the worktree clean for Cleanup.** Violation: the agent creates a commit before completion-review approval, asks a completion-review critic to review a diff that predates the latest applicable repo-root merge, treats a merge that is not `Already up to date.` and not a clean fast-forward as already approved, removes the worktree after such a merge without resetting to the applicable completion-review phase, or removes the worktree while approved implementation changes remain uncommitted. (Text-only: no instrumentation exists to detect whether the latest applicable repo-root state was merged before completion review, whether a merge was non-trivial under this definition, whether the mission reset to completion review after such a merge, or whether worktree removal happened only after fresh approval of the merged state. A test becomes feasible when the harness gains review-phase and merge-state observability.)
5. **Once a `mission.md` has been approved for the active skill, that approved mission is immutable for the rest of that mission. This applies to both `/enhance-harness` and `/new-sdlc`.** If implementation or review reveals the mission must change, the lead agent must stop the current mission, keep the approved `mission.md` unchanged, and follow `.agent/schemas/abort-protocol.md`, including rewriting `handoff.md` to state that the mission is already aborted and not resumable. The lead agent must not autonomously create a replacement mission. Violation: the agent edits an approved `mission.md`, continues execution or review under a modified mission, fails to record the abort state in `handoff.md`, resumes a mission already marked aborted, or drafts a new mission without first stopping and surfacing blockers to the user. (Text-only: no instrumentation exists to detect post-approval `mission.md` edits, whether the agent aborted when an approved mission became invalid, whether the non-resumable abort state was recorded, or whether it autonomously drafted a replacement mission. A test becomes feasible when the harness gains mission lifecycle observability.)
6. **Until a change is merged, the authoritative copies of harness governance artifacts and harness skill-definition files are the main repo-root `CLAUDE.md`, the markdown files under the main repo-root `.agent/schemas/` directory, and the markdown files under the main repo-root `.claude/commands/` directory, not matching files inside a worktree. Agents and critics must resolve those artifacts from the main repo root whenever a harness instruction references them. Matching worktree files and any unstaged changes to them are proposed changes only and are not effective policy before merge. When a review phase already provides raw diff output or repository access that exposes a changed worktree copy of one of those files, the reviewer must inspect the proposed worktree edit as a proposal while still applying the repo-root copy as the current authority until merge. In `/new-sdlc`, the same rule applies to the target repo-root `CLAUDE.md` and to any target-repo skill-definition files that `CLAUDE.md` directs the agent to use, such as `./.claude/commands/*.md`.** Violation: an agent or critic treats a worktree copy of those artifacts as authoritative before merge, fails to resolve them from the applicable repo root, treats unstaged worktree edits to those artifacts as already-effective policy before merge, or ignores a changed worktree copy that was made available through phase-allowed diff output or repository access. (Text-only: no instrumentation exists to detect which copy of `CLAUDE.md`, `.agent/schemas/*.md`, or `.claude/commands/*.md` a live agent or critic resolved during review. A test becomes feasible when the harness gains session-level observability for artifact resolution.)
7. **Invariants are best captured as tests, not text.** Where possible, encode invariants as automated tests rather than prose rules. Text invariants are a last resort when a test is infeasible.
8. **Existing tests are invariants.** Modifying or deleting existing test code is an invariant change. Adding new tests is not. Violation: a commit diff modifies or deletes existing test code without following the invariant-change process (2-critic review, normal flow). (Text-only: no CI or pre-commit hook exists to detect test-file modifications automatically. A test becomes feasible when the harness gains a pre-commit check that flags diffs touching existing test files.)
