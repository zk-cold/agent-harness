# CLAUDE.md

## Invariants

1. **On session start, the lead agent must inspect `.claude/worktrees/` and the main project root before any interview or execution work begins. If `handoff.md` or `mission.md` exists at the main project root, the agent must stop and ask the dev how to proceed before continuing. Otherwise, if exactly one worktree exists and it contains `handoff.md` at its root, the agent must load that `handoff.md` and use it to orient. If multiple worktrees exist, or if exactly one worktree exists but it does not contain `handoff.md`, the agent must stop and ask the dev how to proceed before continuing. If no worktree exists and no legacy root runtime artifacts exist, the agent must create a worktree before any interview or execution work begins. All runtime artifacts (`handoff.md`, `mission.md`) must be written to the worktree root, never the main project root. If a `handoff.md` exists and the agent will treat the current prompt as fresh work instead of resuming that handoff, it must first confirm with the dev before discarding or replacing it.** Violation: the agent begins work on a new request without checking both the worktree directory and the main project root first, continues despite legacy root runtime artifacts or ambiguous worktree state without dev direction, writes `handoff.md` or `mission.md` to the main project root instead of the worktree root, or discards or replaces an existing `handoff.md` while treating the prompt as fresh work without dev confirmation. (Text-only: no instrumentation exists to detect whether the agent checked for a worktree or wrote artifacts to the correct location at session start. A test becomes feasible when the harness gains session-level observability.)
2. **For every work request, the lead agent must invoke `/enhance-harness` or `/new-sdlc` before beginning execution. If the agent cannot determine which skill applies, it must ask the user to disambiguate.** Violation: the agent produces deliverables (files, commits, mission.md) for a work request without having first invoked one of the two routing skills, or proceeds despite ambiguity without asking the user. (Text-only: no instrumentation exists to log skill invocations or assert invocation order; the harness currently has no test infrastructure for agent conversation behavior. A test becomes feasible when the harness gains session-level observability.)
3. **All critic agents must respond with exactly one of: (a) the word "APPROVE" with no additional text, or (b) "REJECT" followed by reasons.** There is no "approval with comments" — if a critic has feedback, it must REJECT. Violation: a critic response contains approval alongside commentary, suggestions, or caveats. (Text-only: no instrumentation exists to parse or validate critic responses programmatically. A test becomes feasible when the harness gains critic-response validation.)
4. **Once a `mission.md` has been approved for the active skill, that approved mission is immutable for the rest of that mission. This applies to both `/enhance-harness` and `/new-sdlc`.** If implementation or review reveals the mission must change, the lead agent must stop the current mission, keep the approved `mission.md` unchanged, and follow `.agent/schemas/abort-protocol.md`, including rewriting `handoff.md` to state that the mission is already aborted and not resumable. The lead agent must not autonomously create a replacement mission. Violation: the agent edits an approved `mission.md`, continues execution or review under a modified mission, fails to record the abort state in `handoff.md`, resumes a mission already marked aborted, or drafts a new mission without first stopping and surfacing blockers to the user. (Text-only: no instrumentation exists to detect post-approval `mission.md` edits, whether the agent aborted when an approved mission became invalid, whether the non-resumable abort state was recorded, or whether it autonomously drafted a replacement mission. A test becomes feasible when the harness gains mission lifecycle observability.)
5. **Invariants are best captured as tests, not text.** Where possible, encode invariants as automated tests rather than prose rules. Text invariants are a last resort when a test is infeasible.
6. **Existing tests are invariants.** Modifying or deleting existing test code is an invariant change. Adding new tests is not. Violation: a commit diff modifies or deletes existing test code without following the invariant-change process (2-critic review, normal flow). (Text-only: no CI or pre-commit hook exists to detect test-file modifications automatically. A test becomes feasible when the harness gains a pre-commit check that flags diffs touching existing test files.)
